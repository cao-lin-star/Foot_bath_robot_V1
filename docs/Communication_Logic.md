# 主控端、桶体端、基站端通信逻辑说明

本文基于当前代码整理通信行为，不包含代码修改。

## 1. 串口角色与周期

| 端口 | 对端 | 用途 | 初始化波特率 |
|---|---|---|---|
| USART1 | Linux 主控端 | 接收主控命令、向主控上报桶体状态 | `UART_Comm_Init()` 中强制为 115200 |
| USART2 | 基站端 | 中转模式下与基站互传帧 | `UART_Comm_Init()` 中强制为 115200 |
| USART3 | 调试日志口 | 只输出日志，不参与协议帧通信 | 115200 |

相关周期：

| 项目 | 当前值 | 代码含义 |
|---|---:|---|
| 通信任务周期 | 50 ms | `Communication_Task` 每 50 ms 调用 `UART_Comm_TaskProcess()` |
| 桶体状态主动上报周期 | 1000 ms | 每秒向 Linux 主控端发送一帧状态 |
| 主控通信超时判断 | 5000 ms | 超过 5 s 未收到 Linux 合法帧则置 `main_timeout_handled=1`；当前停机/清定时逻辑已注释 |
| 基站连接超时判断 | 3000 ms | 3 s 内收到过基站合法帧则认为基站在线 |
| 日志打印周期 | 1000 ms | USART3 输出文本日志 |

## 2. 协议帧格式

所有协议帧固定 30 字节：

| 字节 | 含义 |
|---:|---|
| `frame[0]` | 帧头 1，固定 `0x55` |
| `frame[1]` | 帧头 2，固定 `0xAA` |
| `frame[2]` | 链路模式 |
| `frame[3]` | 命令字 |
| `frame[4]..frame[28]` | 数据区 |
| `frame[29]` | 校验和 |

校验规则：

```text
checksum = (frame[2] + frame[3] + ... + frame[28]) & 0xFF
frame[29] 必须等于 checksum
```

链路模式 `frame[2]`：

| 值 | 名称 | 当前逻辑 |
|---:|---|---|
| `0x01` | 桶体模式 | 主控命令由桶体端执行 |
| `0x02` | 中转模式 | Linux 发来的合法帧都会转发给基站；A 段命令和 `0xB4` 同时由桶体本地解析；基站合法帧会转发给主控 |
| `0x03` | 直连模式 | 认为是合法模式，但当前没有额外直连转发逻辑 |

非法链路模式的 Linux 帧会被丢弃。

## 3. 接收与发送机制

接收：

- USART1 和 USART2 都使用 `HAL_UARTEx_ReceiveToIdle_DMA()`。
- DMA 接收缓冲区长度为 128 字节，循环模式。
- 每次串口 idle/rx event 回调后，代码把新增字节喂给帧解析器。
- 解析器按 `0x55 0xAA` 帧头同步，累计满 30 字节后校验；校验通过才放入待处理槽。

发送：

- USART1 和 USART2 都有独立发送槽。
- 如果当前串口空闲，立即 DMA 发送。
- 如果正在发送，新帧写入 `pending_frame`，等待当前帧发送完成后再发。
- 同一串口忙时多次写入会覆盖 pending，只保留最新一帧。

中断回调：

- `HAL_UARTEx_RxEventCallback()` 调 `UART_Comm_RxEventCallback()`。
- `HAL_UART_TxCpltCallback()` 同时通知协议发送和日志发送。
- `HAL_UART_ErrorCallback()` 会重启 USART1/USART2 接收；USART3 错误由日志模块恢复。

## 4. Linux 主控端到桶体端

Linux 主控端通过 USART1 发送 30 字节协议帧。

处理顺序：

1. 帧头、长度、校验通过后进入 Linux 接收槽。
2. 通信任务取出该帧。
3. 检查 `frame[2]` 是否为合法链路模式。
4. 记录主控通信时间：
   - `last_link_mode = frame[2]`
   - `main_last_rx_tick = HAL_GetTick()`
   - `main_timeout_handled = 0`
5. 根据 `frame[3]` 执行命令。
6. 若命令被桶体端处理，立即向 USART1 回一帧状态。

主控通信连接状态：

- 代码每 50 ms 检查一次是否超过 5 s 未收到 Linux 合法帧。
- 当前超时后的停机、清定时、改状态逻辑被注释，只设置 `main_timeout_handled=1`。
- 上报帧里没有单独的“主控是否在线”字段。

## 5. 桶体端执行的主控命令

命令字 `frame[3]`：

| 命令 | 值 | 桶体端行为 | 是否立即回状态 |
|---|---:|---|---|
| 空闲 | `0x00` | 只记录命令 | 是 |
| 关闭 | `0xA0` | 停止所有输出，清定时，主状态设为关闭 | 是 |
| 待机 | `0xA1` | 停止所有输出，清定时，主状态设为待机 | 是 |
| 开加热 | `0xA2` | 设置目标温度 `frame[6]`，开启加热，水泵循环，运行状态 | 是 |
| 关加热 | `0xA3` | 关闭加热；若 UV 未开则关水泵；运行状态 | 是 |
| 电机 | `0xA4` | 电机档位取 `frame[7]`；档位 0 为待机，否则运行 | 是 |
| UV | `0xA5` | UV 状态取 `frame[8]`；开 UV 时开水泵循环；关 UV 且未加热时关水泵 | 是 |
| 定时 | `0xA6` | 定时编码取 `frame[9]`，同时同步温度/电机/UV/水泵状态 | 是 |
| 停止 | `0xA7` | 停止所有输出，清定时，待机 | 是 |
| 自检 | `0xA8` | 清故障，主状态设为自检 | 是 |
| 低功耗 | `0xA9` | 停止所有输出，主状态设为低功耗 | 是 |
| 调试水泵阀 | `0xB4` | 直接打开水泵+排水阀，运行状态 | 是 |
| 系统复位 | `0xC0` | 停止所有输出，请求 200 ms 后复位 | 是 |

定时编码 `frame[9]`：

| 编码 | 定时时长 |
|---:|---:|
| `1` | 10 分钟 |
| `2` | 15 分钟 |
| `3` | 20 分钟 |
| `4` | 25 分钟 |
| `5` | 30 分钟 |
| 其它 | 清除定时 |

## 6. 主控端到基站端转发

只要满足以下条件，Linux 帧就会从 USART1 转发到 USART2：

```text
frame[2] == 0x02       // 中转模式
```

转发行为：

- 原始 30 字节帧不改内容，直接发给基站端。
- 转发后，桶体端仍继续判断是否需要本地解析。
- A 段命令、空闲命令 `0x00` 和 `0xB4` 调试命令会在桶体本地执行，并立即回一帧桶体状态给 Linux。
- 其它中转命令只转发给基站，不触发桶体状态立即回包；桶体仍会按 1 秒周期主动上报状态。

注意：

- `frame[2] == 0x02` 时，`0xC0..0xCF` 系统命令只转发给基站，不在桶体本地执行。
- `frame[2] != 0x02` 时，桶体端按本地命令逻辑处理 A 段/C 段命令；普通 B 段命令不转发。

## 7. 基站端到主控端转发

基站端通过 USART2 发来的帧，只有在帧自身 `frame[2] == 0x02` 时才会被处理为中转帧。

处理逻辑：

1. 基站帧通过帧头和校验。
2. 检查 `frame[2] == 0x02`。
3. 保存 `frame[16]..frame[28]` 到桶体本地 `base_data[13]`。
4. 记录 `base_last_rx_tick = HAL_GetTick()`。
5. 原始基站帧完整转发给 Linux 主控端。

如果基站帧的 `frame[2]` 不是 `0x02`，当前代码直接忽略，不更新连接时间，也不转发。

## 8. 基站连接状态

基站在线状态通过 `UART_Comm_IsBaseConnected()` 判断：

```text
base_last_rx_tick != 0
且 HAL_GetTick() - base_last_rx_tick < 3000 ms
```

上报给 Linux 的字段：

```text
frame[4] = UART_Comm_IsBaseConnected()
```

含义：

| `frame[4]` | 含义 |
|---:|---|
| `0` | 3 秒内没有收到有效基站中转帧，或从未收到 |
| `1` | 最近 3 秒内收到过有效基站中转帧 |

注意：只有被 `UART_Comm_ProcessBaseFrame()` 接受的基站帧才会刷新连接时间，也就是帧必须合法且 `frame[2] == 0x02`。

## 9. 桶体端主动上报给主控端

桶体端每 1000 ms 主动向 Linux 主控端发送一帧状态；处理主控有效命令后也会立即回一帧状态。

状态帧字段：

| 字节 | 内容 |
|---:|---|
| `frame[0]` | `0x55` |
| `frame[1]` | `0xAA` |
| `frame[2]` | 最近一次 Linux 合法帧的链路模式 `last_link_mode` |
| `frame[3]` | 最近一次命令 `SystemMonitor_GetCommand()` |
| `frame[4]` | 基站连接状态 |
| `frame[5]` | 水位协议值 |
| `frame[6]` | 温度整数值；温度传感器异常时为 0 |
| `frame[7]` | 电机档位 |
| `frame[8]` | UV 状态 |
| `frame[9]` | 剩余定时分钟数 |
| `frame[10]` | 电池电压高字节，单位 0.1 V |
| `frame[11]` | 电池电压低字节，单位 0.1 V |
| `frame[12]` | 主状态 |
| `frame[13]` | 子状态 |
| `frame[14]` | 错误码 1 |
| `frame[15]` | 错误码 2 |
| `frame[16]..frame[28]` | 最近一次有效基站中转帧的数据区副本 `base_data[13]` |
| `frame[29]` | 校验和 |

`frame[9]` 定时分钟数来自 `SystemMonitor_GetTimerRemainingMin()`。当前逻辑为剩余完整分钟，例如：

```text
10:00 -> 10
09:59 -> 9
08:59 -> 8
```

定时真正结束判断使用秒级剩余时间，避免最后 59 秒因为分钟数为 0 而提前关机。

## 10. 运行时数据流总结

主控控制桶体：

```text
Linux 主控 -> USART1 -> 桶体解析 -> 执行 A 段/C 段命令 -> USART1 回状态帧
```

主控透传到基站：

```text
Linux 主控 -> USART1 -> frame[2]=0x02 -> USART2 原帧发给基站；A 段和 0xB4 同时本地解析
```

基站回主控：

```text
基站 -> USART2 -> frame[2]=0x02 的合法帧 -> 更新 base_data/base_last_rx_tick -> USART1 原帧发给 Linux
```

桶体主动上报：

```text
每 1000 ms -> 构建桶体状态帧 -> USART1 发给 Linux
```

日志输出：

```text
每 1000 ms -> USART3 输出文本日志，不参与协议交互
```
